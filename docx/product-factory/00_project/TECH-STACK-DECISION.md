# TECH-STACK-DECISION — Nexa 技术选型决策门（S2.5）

> 项目：**Nexa** —— AI API 大模型网关 SaaS（聚合转发平台，类 OpenRouter / new-api 形态）。
> 阶段：S2.5 技术选型决策门。位置：S2（功能拆分已出 FUNCTION-LIST / MODULE-MAP / NFR）之后，S3 流程设计之前。
> 角色：架构师驱动。本文件给候选 + 权衡 + **架构师明确推荐** + 理由 + 风险；用户拍板后才冻结（本文件**尚未冻结**，见 §8）。
> 硬约束：**架构风格已由用户硬性锁定 = DDD**，本文不讨论"要不要 DDD"，只在 DDD 前提下选语言/框架/落地严格度/库/前端/部署。
> 构建方式：**从零自研**，不二开任何现有代码库。new-api 仅作功能/数据逻辑参考，routifyapi 仅作视觉/营销参考。

## 0. 项目技术特征画像（选型依据）

来自只读上游产物（FUNCTION-LIST.csv / MODULE-MAP.md / NFR-REQUIREMENTS.md / PROJECT.md / 已有 DB-SCHEMA 参考）：

| 维度 | 画像 | 选型含义 |
|---|---|---|
| 系统类型 | AI API 网关 SaaS（数据面转发 + 控制面 SaaS） | 数据面要极致低延迟高并发；控制面是标准重业务 SaaS |
| 规模 | 243 功能 / 13–14 模块 / 22 张表 | 中大型，但单团队可控，**不到必须微服务的体量** |
| 性能 NFR（铁律） | 网关附加延迟 p50≤15ms / p99≤60ms；单实例≥1500 req/s；≥5000 并发长连接（SSE/WS）；选渠 p99≤5ms；预扣+结算 p99≤8ms | **数据面对运行时延迟与并发极度敏感**，是首要选型驱动 |
| 实时性 | 流式响应 SSE 透传 + Realtime WebSocket；异步任务中心 | 必须原生优雅支持 SSE/WS 流式 + 后台任务队列 |
| 复杂域分布 | 复杂核心：**Relay 转发链路 / 计费(预扣-结算-分层-billingexpr DSL) / 渠道选路由(AutoBan/降级/重试/亲和) / 对账用量聚合**；外围 CRUD：账号、令牌配置、用量日志查询、营销页内容 | DDD 严格度**逐子域分档**，不一刀切 |
| 伸缩性 NFR | 转发实例无状态，状态走 Redis，可水平线性扩；缓存命中率≥95%；DB 读写分离；支撑≥10万用户/50万令牌/1000渠道 | 无状态 + 共享缓存 + 关系型主库 + 读副本 |
| 多端前端 | ① 公开营销站（要惊艳 + SEO）② 用户控制台（重交互）③ 管理后台（重表单/表格）④ API 文档站 ⑤ Playground | 营销站需 SSR/SEO；控制台/管理台是 SPA 型重应用 |
| 安全/合规 | Key 加密静态存储、Key 掩码不回显、prompt 脱敏、2FA/Passkey、审计、数据出境合规 | 需成熟加密/WebAuthn/审计生态 |
| 参考系 | new-api = Go 实现（功能/数据逻辑参考）；routifyapi = 视觉参考 | Go 路线有现成逻辑映射，降低领域理解成本（非约束，仅信号） |

**一句话画像**：一个对运行时延迟/并发极度敏感的转发数据面 + 一个标准重业务的 SaaS 控制面，复杂度集中在转发/计费/选路/对账四个核心子域，需要多端前端且公开站要惊艳。

## 1. 维度一：后端语言 / 框架

> 评估轴：高并发低延迟（NFR 首要）、对 DDD 分层与充血模型的承载、流式 SSE/WS 原生支持、生态成熟度、开发与维护速度、单团队可持续性。

### 候选

| 候选 | 优势 | 劣势 | 对 DDD 支持 | 对本项目契合度 |
|---|---|---|---|---|
| **A. Go（Gin/Echo + 手搓 DDD 分层）** ⭐推荐 | goroutine 天生适配高并发长连接（5000+ SSE/WS 轻松）；GC 低延迟、内存占用小；编译单二进制部署极简；SSE/流式透传是 Go 强项；new-api 本就是 Go，领域逻辑可直接映射理解 | 无内建 DDD 框架，分层/聚合/领域事件需手工约定（靠规范不靠框架强制）；泛型/表达力弱于 Java/Kotlin；ORM(GORM) 偏数据驱动，写充血模型要刻意克制 | 中（靠团队纪律落地，非框架兜底）；轻量~战术完整 DDD 完全可行 | **极高**：性能命门正中 Go 强项，部署运维最省，参考系同语言 |
| B. Java 21 / Kotlin（Spring Boot 3 + Spring Modulith） | DDD 生态最成熟（聚合/领域事件/Modulith 模块边界强制）；虚拟线程(Loom)后高并发已大幅改善；生态/中间件/招聘面最广 | JVM 内存占用与冷启动重；低延迟需调 GC/Loom 调优，p99≤60ms 网关附加延迟需更仔细工程；框架重、样板多，小团队迭代偏慢 | 高（Modulith 强制模块边界 + 充血模型 + 领域事件开箱） | 中高：DDD 最顺，但延迟/资源代价大于 Go，转发数据面非其最优场景 |
| C. Rust（Axum + tokio） | 极致性能与最低延迟、零 GC 抖动，理论最契合网关数据面；内存安全 | 开发速度最慢、领域建模/业务迭代成本高；生态（支付/2FA/审计/ORM）相对不成熟；招聘/维护门槛最高；243 功能的业务面用 Rust 开发周期不划算 | 中（类型系统利于建模，但业务迭代慢） | 中：性能过剩而开发成本失衡，过度工程风险高 |

### 架构师推荐：**A. Go（Gin 或 Echo + 手工 DDD 分层）**

**推荐理由**：
1. **性能命门正中靶心**。NFR 第一优先级是网关附加延迟（ms 级）+ 5000 并发长连接 + 1500 req/s/实例。Go 的 goroutine 模型 + 低延迟 GC + SSE/流式透传是它的主场，无需为达标做重度调优。Java 需 Loom + GC 调参才追平，Rust 性能过剩但开发太慢。
2. **DDD 可优雅承载（务实档）**。本项目 DDD 定位是"轻量打底 + 核心域战术完整"，并不需要 Spring Modulith 那种框架级强制。Go 用目录分层 + 接口约定 + 充血实体完全能落地 DDD，团队纪律 + 代码评审兜住即可（§5 编码硬约束传给 backend-engineer）。
3. **部署运维最省**。单静态二进制 + 极小镜像，水平扩展、冷启动、资源占用全面优于 JVM，直接支撑无状态横扩 NFR-E01。
4. **参考系同语言降风险**。new-api 是 Go，转发/选渠/计费/对账的领域逻辑可直接对照理解（注意：是参考逻辑，不二开代码），大幅降低核心域建模的认知成本。

**唯一需向用户点明的取舍**：选 Go = 用"团队纪律 + 代码规范"来落地 DDD，而非靠框架强制。如果用户更看重"框架硬性兜住架构不走样、且团队是 Java 背景"，可改选 Java/Kotlin + Spring Modulith（代价：延迟/资源开销更大、迭代更慢）。这是本维度**唯一需要用户拍板的核心取舍**。

## 2. 维度二：DDD 落地严格度（逐子域分档）

> 策略：轻量级 DDD 打底全项目（四层分层 + 充血领域模型 + repository 接口抽象），**战术完整 DDD 局部用在真正复杂的核心子域**，外围 CRUD 保持轻量。对照 MODULE-MAP 逐模块标注。**默认模块化单体内的逻辑分层，DDD ≠ 微服务。**

| 模块/子域（对照 MODULE-MAP） | 复杂度 | 推荐档位 | 理由 |
|---|---|---|---|
| **Relay 网关**（转发链路 F-3026~） | 极高（性能+状态机+流式） | **战术完整**：选渠/重试/降级建为领域服务 + 转发上下文聚合 + 领域事件（请求成功/失败/切渠） | 核心命门，逻辑最重，需清晰聚合边界 |
| **计费与钱包**（F-2038~F-2048，含预扣/结算/分层/billingexpr） | 极高（金额强一致） | **战术完整**：Wallet/Quota 聚合根 + 金额值对象 + 预扣-结算领域事件 + billing 表达式作为领域服务；聚合内强一致 | 涉及钱，一致性要求最高，最该上完整战术 DDD |
| **渠道路由(管理)**（F-2016~，AutoBan/亲和/分组） | 高 | **战术完整**：Channel 聚合根 + 健康状态值对象 + Ability 关系 + 状态流转领域事件 | 选路质量直接决定可用性，状态机复杂 |
| **异步任务中心**（F-2001~F-2015） | 中高 | **战术完整（轻）**：Task 聚合根 + 状态机 + 领域事件驱动队列 | 状态流转 + 异步编排有真实复杂度 |
| 对账 / 用量聚合（usedata，DC-007 计费数据） | 高 | **战术完整（CQRS 读模型）**：写侧走日志事件，读侧建独立聚合统计模型 | 写读分离利于准实时聚合 + 历史可追溯 |
| 账号与身份（F-1001~F-1054，OAuth/2FA/Passkey） | 中 | **轻量级**：充血 User 实体 + repository，认证逻辑领域服务 | 多但模式化，无需聚合根重武器 |
| 控制台-令牌（F-3001~） | 低中 | **轻量级**：充血 Token 实体 + repository | CRUD + 作用域规则为主 |
| 控制台-用量日志（F-4001~） | 低 | **轻量级（偏读）**：查询走读模型/DTO，不强套领域 | 主要是查询/导出 |
| 增长(签到/分销 F-1039~) | 低中 | **轻量级**：充血实体 + 返利领域服务 | 规则集中但不复杂 |
| 模型广场（F-3013~） | 低 | **轻量级**：展示/映射配置 CRUD | 配置型 |
| 公开与营销（F-4039~F-4044） | 低 | **轻量（几乎无领域）**：内容/CMS 配置读取 | 纯展示 |
| 横切-NFR/合规/RBAC（F-5001~） | — | **基础设施层 + 横切关注点**（中间件/拦截器/审计），不建独立领域聚合 | 限流/审计/指标是横切，不是领域 |

**推荐总结**：全项目**轻量级 DDD 打底**；**Relay / 计费 / 渠道路由 / 异步任务 / 对账聚合** 五个核心子域上**战术完整 DDD**（聚合根/值对象/领域事件）；其余 CRUD/展示子域保持轻量。避免对营销页、用量日志这种外围也套聚合根（过度工程，违反 §8 Pitfall）。

**充血模型硬要求**（传递给 backend-engineer）：领域行为写在实体/聚合上，禁止"贫血 DTO + service 堆逻辑"冒充 DDD。

## 3. 维度三：数据库 / 缓存 / 消息队列 / 分析库

### 3.1 主库（关系型）

| 候选 | 权衡 |
|---|---|
| **PostgreSQL** ⭐推荐 | 功能最全（JSONB、分区表、丰富索引、强一致），读写分离/分区/归档（NFR-E03/DC-006）成熟，计费/对账强一致场景首选 |
| MySQL 8 | 同样可读写分离，运维熟悉度高；功能略逊 PG（JSON/分区/CTE 弱一些） |
| 三库兼容(SQLite/MySQL/PG) | new-api 走的路线，利于私有化轻量部署；但从零自研、目标是 SaaS 托管，三库兼容会持续束缚 SQL 表达力与索引优化，不建议作为强制目标 |

**推荐：PostgreSQL 作为唯一主库**（SaaS 托管自研，无需为他人私有化兼容三库；PG 的分区/JSONB/强一致最契合计费对账与日志归档）。日志/用量等大表用 **PG 原生分区**（NFR-E03 日志分区/归档）。生产读写分离用主从 + 读副本。

### 3.2 缓存

**推荐：Redis（必选）**。理由：NFR-E01 无状态横扩要求选渠/会话/亲和状态走共享缓存；NFR-E02 缓存命中率≥95%、DB 选渠查询占比<5%。配 **进程内内存缓存做只读回落**（NFR-A07：Redis 失联时回落内存继续选渠）。这是双缓存双路径，非可选项。

### 3.3 消息队列 / 异步任务中心

| 候选 | 权衡 |
|---|---|
| **Redis Streams / Asynq（基于 Redis）** ⭐推荐（起步） | 已有 Redis，零新增中间件；Asynq(Go) 提供重试/延时/调度，足够支撑异步任务中心(F-2001~)、用量聚合、对账批处理、通知告警；运维最省 |
| NATS / NATS JetStream | 轻量高性能消息总线，适合领域事件解耦；比 Kafka 轻，比 Redis Streams 更专业 |
| Kafka / RabbitMQ | 功能强但运维重；当前规模(10万用户级)用不上 Kafka 的吞吐，徒增复杂度 |

**推荐：起步用 Redis + Asynq 做异步任务中心与领域事件投递**（任务/通知/聚合/对账批处理）。理由：避免一上来引入 Kafka 这类重型中间件（违反 §8 Pitfall 过度工程）。**预留演进路径**：若后续领域事件量/解耦需求增长，再升级到 NATS JetStream。**不引入 Kafka**，除非用户明确要求事件溯源级别的持久日志。

### 3.4 时序 / 分析库（用量统计）

| 候选 | 权衡 |
|---|---|
| **起步：PG 物化视图 / 预聚合表（QuotaData 按日聚合）** ⭐推荐 | NFR-O07 用量数据准实时（≤5min 聚合）用 PG 预聚合 + 物化视图即可达标；不新增组件 |
| 演进：ClickHouse | 当调用日志量级到数亿行、多维实时分析/排行榜成为瓶颈时再引入做 OLAP 用量分析库 |

**推荐：起步不上独立分析库**，用 PG 预聚合表 + 物化视图承接 NFR-O07 准实时用量。**在 DECISION 里预留 ClickHouse 演进位**：日志量爆发后将「用量分析/排行榜」读路径迁到 ClickHouse（CQRS 读侧天然适配）。

**数据层推荐总结**：PostgreSQL（主库，大表分区）+ Redis（缓存，必选）+ 进程内内存缓存（只读回落）+ Redis/Asynq（异步任务中心，起步）+ PG 预聚合（用量统计，起步）。ClickHouse / NATS 作为明确的演进预留，非首发引入。

## 4. 维度四：前端框架

> 难点：① 公开营销站要**惊艳 + SEO**（SSR/SSG）② 控制台是重交互 SPA ③ 管理后台重表单/表格 ④ API 文档站 ⑤ Playground。需在"营销站要 SEO"与"控制台/管理台是富应用"间取平衡。

### 候选

| 候选 | 权衡 |
|---|---|
| **A. Next.js (React) 单技术栈分应用** ⭐推荐 | 营销站用 Next SSR/SSG（SEO + 首屏惊艳）；控制台/管理台用 Next（或纯 React SPA）；React 生态/组件库(antd/shadcn/Tailwind)/招聘面最广；一套 React 心智覆盖全部多端，组件/设计系统可复用 |
| B. Nuxt (Vue) | 同样 SSR + SEO，Vue 上手快、模板友好；生态略逊 React，复杂控制台的高级组件与人力市场不及 React |
| C. 营销站(Astro/Next) + 管理台(纯 Vite+React SPA) 拆栈 | 营销站用 Astro 极致 SEO/性能，后台用轻 SPA；优点是营销站最快最轻，缺点是两套栈/两套设计系统，维护成本上升 |

### 架构师推荐：**A. 全栈统一 React，按应用形态分构建**

具体落地：
- **公开营销站 + API 文档站**：**Next.js（SSR/SSG）** —— 拿 SEO 与首屏惊艳（routifyapi 视觉参考要落地需要强表现层）。文档站可用 Next + MDX 或 Nextra。
- **用户控制台 + 管理后台 + Playground**：**React SPA**（Vite + React，或同 Next 的 client 区），鉴权后富交互，无 SEO 需求。组件库走 **Tailwind + shadcn/ui（或 antd）**，控制台/管理台共用一套设计系统。

**推荐理由**：
1. **一套 React 心智覆盖全部五端**，组件库/设计 token/工程链统一，最低维护成本；招聘与生态最广。
2. **营销站 SSR/SSG 拿 SEO 与惊艳首屏**（Next 是该场景事实标准），同时控制台用 SPA 模式获得富应用体验，鱼与熊掌兼得。
3. 比"拆 Astro + 独立 SPA"少一套技术栈，降低多端设计系统割裂风险。

**需向用户点明的取舍**：营销站若追求**极致 Lighthouse 性能/最轻包体**，Astro 比 Next 更优，但代价是多维护一套栈。本推荐选"统一 React 省维护"，若用户更在意营销站极限性能可改 C 方案。

## 5. 维度五：部署形态 + 架构拓扑 + CI/CD

### 5.1 单体 vs 微服务

| 候选 | 权衡 |
|---|---|
| **模块化单体（Modular Monolith）** ⭐推荐 | DDD 子域 = 模块内逻辑边界（不拆进程）；单部署单元、本地调用零网络开销（利于网关 ms 级延迟）；运维最省；243 功能/单团队完全可控。**DDD ≠ 微服务**，这是务实首选 |
| 微服务（拆数据面/控制面/计费等） | 仅在团队规模、独立伸缩、独立发布有强需求时才值得；当前会徒增网络延迟（违背 ms 级 NFR）+ 分布式事务复杂度(计费一致性) + 运维负担 |

**推荐：模块化单体**。子域按 §2 在代码内分模块/分层，**逻辑解耦但同进程部署**。理由：网关数据面对延迟敏感，进程内调用 > 跨服务网络跳；计费强一致在单库单进程内最易保证；单团队运维负担最低。

**唯一可考虑的物理拆分**：若未来数据面吞吐压力极大，可把**无状态 Relay 转发实例**与**控制面（控制台/管理台 API）**部署为**同一代码库的两种运行模式/两组实例**（数据面多副本横扩，控制面少副本），共享库与领域代码——这是"一个单体、两种部署角色"，非微服务拆分。建议作为演进选项写入。

### 5.2 容器化 + 编排

- **容器化：Docker**（Go 单二进制 → 极小镜像）。✅推荐。
- **编排**：起步 **Docker Compose / 单机或少节点**即可（无状态实例 + 共享 Redis/PG）；规模上来后迁 **Kubernetes**（无状态实例天然适配 HPA 水平扩，对接 NFR-E01/E05 弹性限流）。**不要一上来就上 K8s**（过度运维）。

### 5.3 CI/CD

- **CI**：GitHub Actions（或 GitLab CI）—— 编译 + 单测 + 机械门脚本（项目已有 product-quality-gates 机械门习惯）+ go vet/lint + 前端构建。
- **CD**：构建镜像 → 推镜像仓库 → 滚动发布无状态实例（蓝绿/滚动，配合健康探针 NFR-O06 liveness/readiness）。
- **可观测性栈**（呼应 NFR-O）：Prometheus（/metrics RED 指标）+ Grafana + OpenTelemetry（trace_id 链路）+ 多渠道告警（Email/Webhook/Bark）。

**部署推荐总结**：Go 单二进制 → Docker → 起步 Compose/少节点、演进 K8s；模块化单体（可演进为"一库两运行角色"：数据面/控制面分组实例）；GitHub Actions CI/CD；Prometheus+Grafana+OTel 可观测性。

## 7. 风险与缓解

| 风险 | 等级 | 缓解 |
|---|---|---|
| Go 无框架强制 DDD，团队可能写成贫血模型/假 DDD | 中高 | 编码硬约束（充血模型必须）写入 backend-engineer 角色 + 代码评审门 + 目录分层模板冻结 |

| **主数据库** | **PostgreSQL 单库**（放弃三库兼容） | 从零自研、SaaS 托管，单库换 SQL 表达力与优化空间；大表分区 + 读写分离演进 |

| **用量统计** | 起步 **PG 预聚合（CQRS 读模型）**，预留 ClickHouse 演进位 | 读侧抽象，迁移只换读侧实现 |

**关键工程纪律（冻结附带，传递给 backend-engineer / S8）**：
- 虚拟线程避 pinning：用 `ReentrantLock` 代替 `synchronized` 做阻塞临界区（Java 21-23；Java 24 已缓解）。
- 显式 ZGC，容器内存上限给足别用 K8s 默认小 limit 卡死 JVM。
- 虚拟线程下不用固定线程池限流，并发控制用 Semaphore/限流器。
- SSE 长连接设读写超时 + 空闲检测 + 上游慢时背压。

**下游连带修订（必须执行）**：
- ⚠️ **S7 开发契约的 DB-SCHEMA 需修订**：原版假设 Go + GORM + 三库兼容，现改为 **Java + JPA/MyBatis（待 S7 内定 ORM）+ PostgreSQL 单库**。表结构逻辑不变，但 ORM model 表达从 GORM struct 改为 JPA Entity；放弃三库兼容后可用 PG 专有特性（JSONB、分区、部分索引等）。此修订在进 S8 前由架构师在 S7 层处理（DB-SCHEMA + MIGRATION-PLAN 重新生成），openapi.yaml/API-ENDPOINTS/BACKLOG/覆盖校验不受影响（与语言无关）。
