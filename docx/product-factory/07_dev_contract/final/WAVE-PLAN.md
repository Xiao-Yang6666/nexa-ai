## W1 — 基础设施 + 鉴权 + 账号身份（66 task）

**模块**：部署管理（F-3039~F-3056）、账号与身份 D1（F-1001~F-1054 核心鉴权部分）、RBAC（F-5030~F-5035）、Security（F-5018~F-5021）。

**为什么先行**：所有业务模块都依赖「用户已登录 + 角色鉴权 + 部署环境就绪」。User/Token 表是几乎所有外键的被依赖方（见 MIGRATION-PLAN 依赖分层第 1 层）。

**关键依赖链**：
- 部署/环境（io.net 集成、AutoMigrate 起表）→ 鉴权中间件（UserAuth/AdminAuth/RootAuth）→ 注册/登录/OAuth/2FA/Passkey → 个人信息/用户管理。
- RBAC 角色矩阵在此 wave 落地（中间件级），后续 wave 所有端点的 `security` 直接复用。

**前后端并行点**：登录/注册/OAuth 端点契约冻结后，前端登录滑出抽屉（S6 已设计）可立即对接 mock；后端并行实现真鉴权。

---

## W2 — 数据层核心域（72 task）

**模块**：渠道管理与上游路由 + 渠道运维 + D7（F-2016~F-2037、F-6005/6006）、模型元数据/同步/供应商/D8/模型广场（F-3013~F-3025、F-6001~6004）、令牌管理（F-3001~F-3012）、计费与额度 D6（F-2038~F-2048、F-6007~6009）、亲和缓存（F-2029~F-2033）、跨分组重试（F-2034~F-2037）、预填分组（F-2012~2015）。

**为什么第二波**：这是网关的「配置面」——渠道（上游 B）、对外模型（A）、两层映射（C→A→B）、令牌、计费倍率，全是 W3 实际转发时要读的基础数据。必须先有渠道和模型目录，relay 才有东西可转。

**关键依赖链**：
- PublicModel(A) + Channel(B) → PlatformModelMapping(A→B) + ChannelModelCost(channel×B) → UserModelAlias(C→A) → 计费倍率配置。
- 令牌依赖 User（W1）+ 分组折扣（计费）。
- 预填分组、亲和缓存配置是渠道/令牌配置的辅助，同波。

**前后端并行点**：models-admin / channels / groups / model-map / keys / profit 等管理台页面（S6 已出原型）照 openapi mock 先行；后端并行实现 A→B 映射解析、渠道池容灾、成本配置。**客户视图 DTO 裁剪（无 B/成本/利润）在此 wave 的响应实现中钉死**。

---

## W3 — 业务转发（46 task）

**模块**：Relay 网关 + D9 中继转发（F-3026~、F-6010 协议兼容/F-6011 端到端链路）、异步任务中心（F-2001~F-2015）、日志与用量 + D5（F-4001~F-4014、F-6012 调用明细）、Playground（F-4038）、Observability（F-5022~F-5024）。

**为什么第三波**：转发链路依赖 W2 的渠道+模型映射+计费全部就绪——一次 /v1/chat/completions 要走「C→A 解析（UserModelAlias）→ A→B 选渠（PlatformModelMapping + 渠道池）→ 协议兼容（adapter）→ 上游调用 → 计费（quota_sell/cost/profit）→ 日志落库」。

**关键依赖链**：
- relay 入口（协议兼容层，OpenAI⇄Anthropic）→ 映射解析（依 W2）→ 选渠重试（依 W2 亲和/跨组）→ 计费结算（依 W2 倍率）→ Log 写入（10 新字段）。
- 异步任务（视频/音乐/MJ）附着 relay 链路，任务状态机 CAS + 退款结算。
- 日志用量读 Log 表：**UserLogView（本人，无 B/成本/利润）vs AdminLogView（全链 C→A→B+成本利润）两套 DTO**。

**前后端并行点**：usage / logs / tasks 页面照 mock 先行；后端并行实现 relay 链路。relay /v1/* 是对外标准 OpenAI/Anthropic 协议，外部客户端（SDK）不感知内部映射。

---

## W4 — 增长 + 对外页 + 模型映射前台（22 task）

**模块**：签到 D2（F-1046~F-1050）、邀请返利分销 D3（F-1039~F-1045）、Telegram 登录 Bot D4（F-1051~F-1054）、公开站点（F-4039~F-4044）、排行榜快照（F-4010）、客户层模型映射前台（F-6003 候选联想）。

**为什么第四波**：增长功能依赖 W1 账号 + W3 用量数据（排行榜读用量快照、分销读消费返利）；公开站排行/定价页依赖 W2 模型目录 + W3 用量聚合。属于「锦上添花」层，可在核心闭环跑通后并行做。

**前后端并行点**：home/models/pricing/ranking/checkin/referral 页面（S6 全部出过原型）照 mock 先行；后端并行实现签到日历结算、分销返利计算、排行榜定时快照、Telegram bot webhook。

---

## W5 — 运维后台 + 横切收尾（37 task）

**模块**：运营与运维（F-4015~F-4037：系统设置/选项/性能监控/审计）、NFR（F-5001~F-5005）、Compliance（F-5006~F-5011）、Scalability（F-5012~F-5017）。

**为什么最后**：运维监控/审计需要业务跑起来才有数据可监控；横切约束（限流/HTTPS/水平扩展/审计埋点/数据合规）是中间件/基础设施层，随业务模块落地时逐步嵌入，最后统一收口验收。NFR/合规/扩展性多为「无独立端点的横切约束」，由部署+中间件实现，在此 wave 集中验收。

**关键点**：
- 审计日志埋点（系统级）在前序各 wave 写业务时已埋，W5 验收完整性。
- 限流/敏感词/自动分组/i18n 等运维配置项收口。
- NFR/Scalability/Compliance 横切项做最终合规验收（对照 NFR-REQUIREMENTS.md）。

## 前后端并行总策略

```
                    ┌─────────────── API 契约冻结（openapi.yaml）───────────────┐
                    │                                                          │
   后端 (S8)        │  W1 鉴权→W2 数据层→W3 转发→W4 增长→W5 运维             │
   照 DB-SCHEMA      │       照各 wave 依赖顺序串行/小并行实现端点              │
   + openapi 实现    │                                                          │
                    │                                                          │
   前端 (S9)        │  各 wave 页面照 openapi mock + S5/S6 原型并行对接        │
   照 openapi mock   │  （前端不等后端真实现，靠 mock 自测，契约一致即可对接）  │
   + S6 设计         │                                                          │
                    └──────────────────────────────────────────────────────────┘
```

**冻结纪律**：openapi.yaml 一旦在 S7 冻结，S8/S9 不准私改契约。如需改 → 回 S7 重冻结 + 重跑覆盖校验，避免前后端各改一半对不上。

**并行解耦验证**：前端用 openapi mock server（如 prism）起桩，后端用契约测试（contract test）对 openapi 验证响应结构，两侧各自对契约负责，集成时自然对齐。
