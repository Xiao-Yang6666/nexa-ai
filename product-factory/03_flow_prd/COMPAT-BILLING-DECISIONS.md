# 兼容层 + 计费体系 — 用户拍板的业务规则定稿（2026-06，唯一权威）

> 本文件是用户与产品多轮对齐后拍板的业务规则。架构师 / 产品智能体设计时，**以本文件为准**，
> 任何设计不得违反这里的规则；如发现规则间冲突，回到本文件，不要自行发挥。

## 0. 一句话定位
给现有 new-api 网关加一层「兼容 + 经营」能力：对外一个 key 通吃所有端点和模型、客户端零改动；
对内成本/售价分离、每笔利润算得清、按等级折扣售卖。

---

## 1. 协议兼容层（一个 key 通吃端点）
- 客户用 **OpenAI 格式** 或 **Anthropic 格式（/v1/messages）** 打进来都接（这次只做这两个，架构留扩展位给 Gemini/embedding/图像/语音）。
- 底层按"最终目标模型供应商要求的协议"去调，响应转回客户入站协议。
- **只看头和尾**：协议转换只在"确定最终目标模型后"做一次（入站协议 vs 目标供应商协议；相等直通，不等转换）。中间模型映射跳几层都不影响协议转换。
- 架构基线：复用 new-api 现有 `relay/channel/adapter.go` 的 ConvertOpenAIRequest/ConvertClaudeRequest + types.RelayFormat，抽象成可插拔协议适配器 + IR 中间表示。

## 2. 两层模型映射（C → A → B，链式，1对1）
- **C = 客户别名**（客户请求里的 model 值）。
- **A = 平台公开名**（公开模型列表里、客户可见、用于定价）。
- **B = 真实上游模型**（超管底仓实际调用名，**客户永不可见**）。
- 执行顺序固定：**先客户层 C→A，再超管层 A→B**，最终调用 B。两层都是 1对1 纯字符串替换。
- **客户层映射（C→A）**：客户自助配，作用域 = 分组级 + 用户级（用户级优先于分组级）。
  - 目标 A 的输入是「开放输入 + 候选提示」：输入 GPT 联想出候选（候选来源 = 平台公开模型列表），
    但**不强制白名单**，客户硬输平台没有的名字也允许存（调用时可能失败，那是客户的事），可给浅黄提示但不拦。
- **超管层映射（A→B）**：超管配的全局底仓映射，客户全程看不到 B。
- **防循环**：链式解析要有环检测 + 最大跳数限制。
- **B 不可见三道闸**（架构已设计，保留）：数据层无客户读 B 接口 / 序列化层客户视图只给 C→A、超管视图给全链 / 候选层只返平台公开模型 A 全集。

## 3. 模型分级（品质拆分 —— 关键决策）
- **品质/体验不同的，拆成不同的对外模型分开卖、分别定价**。
  例：opus-4.8（满血）、opus-4.8-max、opus-4.8-air（残血/经济版）= 三个对外模型、三个价。
- **品质一样、只是不同供应商进货价有高低的，才混在同一个对外模型下做多供应商兜底**。
- 红线：品质不同的渠道**绝不**混在同一个对外模型下兜底（否则客户花满血钱拿残血货 = 投诉 + 成本算不清）。

## 4. 成本 / 售价分离（补上的核心漏洞）
- **售价**：挂在「对外模型」上，是公开站展示的**基准价**（模型级定价）。对客户恒定，不管内部走了哪个供应商。
- **成本**：挂在「供应商渠道 × 真实模型 B」上。同一模型不同供应商成本不同，各记各的。
- **每笔请求**：实际走了哪个供应商，就用那个供应商的成本记账；收客户钱按"对外模型基准价 × 分组折扣系数"。
- **利润 = 售价 − 成本**，逐笔可算、可按 模型/供应商/分组 聚合。
- **兜底切供应商时售价恒定**（客户付的钱只看对外模型 + 分组等级，不随内部切换波动）；成本跟实际供应商走。
- **成本倍率录入方式**：A 起步 —— 超管在后台**手动填**每个供应商每个模型的成本倍率（以后再考虑"填进货单价自动折算"）。

## 5. 模型权限：全开（关键决策 —— 砍掉等级控权限）
- **所有对外模型对所有客户可见可用**，不再用分组圈定"能用哪些模型"。
  （中转站通行做法：模型敞开，只按付费等级给不同折扣。）
- 因此**不存在"客户自我授权越权"问题**：模型本来全开，客户自由映射/自选 key 范围都安全。

## 6. 分组：纯折扣等级（不再定价、不再控权限）
- 分组退化为「折扣系数」单位：free=1.0 / vip=0.85 / svip=0.7（示例，具体值后台可配）。
- 升级方式：按充值额 / 付费包等级自动升档。
- **最终扣费 = 对外模型基准价 × 分组折扣系数**。
- 公开站展示基准价；客户登录后按自己等级看折后价。

## 7. key 级：客户自我约束（可选，纯省心，非权限闸门）
- 客户建 key 时**可选**限定"这个 key 只能用某些模型 / 某些端点"——纯粹给自己上保险（防泄露爆量 / 限制某项目只用便宜模型）。
- 默认不选 = 全部可用。这是"做减法的自我约束"，不是"做加法的自我授权"（因为模型本就全开，无加法可做）。
- 端点同理：key 可选限定能用哪些端点协议，默认全开。

## 8. 统计 / 日志（必须）
- Log 扩展字段记录三段式 + 协议 + 成本售价：requested_model(C) / resolved_public_model(A) /
  actual_upstream_model(B) / inbound_protocol / upstream_protocol / protocol_converted /
  以及该笔的 售价金额 / 成本金额 / 利润。
- 客户日志视图：只显示 C→A（看不到 B、看不到成本）。
- 超管日志视图：显示全链 C→A→B + 实际供应商 + 成本 + 售价 + 利润。

## 9. 配套后台能力（用户的核心动机）
- **利润分析看板（admin）**：按 模型 / 供应商 / 分组 看利润，识别"哪个供应商在亏、该给谁多导流量、哪个对外模型定价偏低"。
- **成本/售价配置页（admin）**：供应商成本倍率 与 对外模型基准售价 分两处配，改完即时生效。

## 10. 完整一笔请求链路（end-to-end）
```
客户(分组=vip, key) 用 OpenAI 格式发 model=C
 ① 协议识别：inbound = OpenAI
 ② 模型映射：C →(客户层)→ A →(超管层)→ B   （B 客户不可见）
 ③ key 级自我约束校验：A 在该 key 允许范围内？（默认全开则放行）
 ④ 路由：B 在多个供应商渠道间按优先级/权重选一个（如选中 Y），挂了兜底切下一个
 ⑤ 协议头尾对比：inbound(OpenAI) vs Y供应商协议(如 Claude) → 不等则转换
 ⑥ 扣客户：对外模型 A 的基准售价 × vip 折扣系数
 ⑦ 记成本：供应商 Y 对 真实模型 B 的成本倍率
 ⑧ 响应：按 inbound(OpenAI) 协议转回客户
 ⑨ 落 Log：C/A/B + 实际供应商Y + 协议转换标记 + 售价/成本/利润
```

## 不做（明确排除，避免镀金）
- 不做"按等级控制模型可用性"（模型全开）。
- 不做映射层负载/容灾（负载兜底交给现有 Channel/Ability 路由）。
- 这次协议只做 OpenAI ⇄ Anthropic（其余留扩展位，不实现）。
- 成本录入这次只做手填倍率（不做自动折算）。
